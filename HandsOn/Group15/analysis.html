1. Analysis of the dataset

We will work with three Madrid smart-city datasets that are reusable across the course:

  1. Air Quality – Daily data since 2001 (Ayto. de Madrid).
    URL: Calidad del aire. Datos diarios desde 2001
    Schema (CSV yearly): station_code, date, pollutant, value, unit.
  2. BiciMAD – Stations (Ayto./EMT).
    URL: Estaciones de Bicimad (CSV)
    Schema: station_id, name, address, lat, lon, capacity, status.
  3. Sports facilities – Municipal basic facilities (Ayto. de Madrid).
    URL: Instalaciones Deportivas Básicas Municipales
    Schema: facility_id, name, address, lat, lon, type.

Linkability: all three include location (coordinates/address), enabling links to neighborhoods/districts or external gazetteers. Temporal coverage: air quality provides long-term time series.

3. Resource Naming Strategy

We adopt HTTP slash URIs for scalability, caching and clarity. Base namespace:

https://example.org/group15/

3.1 URI patterns

Collections & resources (identifiers do NOT contain "/" character):

  • Ontology terms: https://example.org/group15/def/{term}
    Example: https://example.org/group15/def/AirQualityStation
  • Air quality stations: https://example.org/group15/station/air/{stationId}
    Example: https://example.org/group15/station/air/47
  • BiciMAD stations: https://example.org/group15/station/bike/{stationId}
    Example: https://example.org/group15/station/bike/123
  • Sports facilities: https://example.org/group15/facility/{facilityId}
    Example: https://example.org/group15/facility/456
  • Measurements (daily): https://example.org/group15/measurement/air/{stationId}-{yyyymmdd}-{pollutant}
    Example: https://example.org/group15/measurement/air/47-20250920-NO2
    Note: Date format is compact (yyyymmdd) with hyphens separating components, no slashes in identifier fragment.

3.2 Design rationale

  • Slash URIs: Better for HTTP caching, clear hierarchical structure, standard practice.
  • No slashes in identifiers: The identifier fragment (after the last collection slash) uses hyphens to separate components, ensuring clean URI structure.
  • Opaque vs descriptive: Station IDs are opaque (from source data); measurements use compound descriptive identifiers for human readability.
  • Date format: Compact ISO-like format (yyyymmdd) enables lexicographic sorting and avoids excessive URI depth.

3.3 Persistence policy

URIs are designed to be stable and persistent:

  • Identifiers will not change once assigned.
  • If resources must be moved or reorganized, we will implement HTTP 301/302 redirections.
  • Deprecated resources will return HTTP 410 Gone with appropriate metadata.
  • Content negotiation via Accept headers will provide HTML (for humans) and RDF formats (Turtle, RDF/XML, JSON-LD) for machines.

4. Ontology design

We provide a lightweight ontology reusing standard vocabularies where possible:

4.1 Reused vocabularies

  • Schema.org: schema:address, schema:City
  • WGS84 Geo: geo:lat, geo:long
  • OWL: owl:sameAs for identity links
  • XSD: datatypes (xsd:dateTime, xsd:decimal, xsd:string)

4.2 Core classes

  • ex:Facility – Generic urban facility or station (superclass)
  • ex:Station – Physical station with geolocation (subclass of Facility)
  • ex:AirQualityStation – Station collecting air pollutant data (subclass of Station)
  • ex:BicycleStation – Bike-sharing docking station (subclass of Station)
  • ex:SportsFacility – Municipal sports facility (subclass of Facility)
  • ex:Measurement – Single observed value at a timestamp

4.3 Properties

Object properties:

  • ex:hasMeasurement – Links a station to measurements observed there

Datatype properties:

  • ex:stationId – Station identifier (string)
  • ex:observedValue – Numeric measurement value (decimal)
  • ex:unit – Measurement unit (string, e.g., "µg/m³")
  • ex:observedProperty – Property being measured (string, e.g., "NO2", "PM10")
  • ex:dateTime – Timestamp of measurement (xsd:dateTime)

5. Data quality & completeness

  • Coverage: Example data includes 5 air quality stations, 5 bicycle stations, 5 sports facilities, and 20+ measurements spanning multiple dates.
  • Validation: All RDF validates against the ontology schema.
  • Geolocation: All facilities/stations include lat/long coordinates for spatial queries.
  • Temporal data: Measurements include proper xsd:dateTime timestamps enabling time-series analysis.

6. Future extensions

  • Link to external gazetteers (GeoNames, DBpedia) for district/neighborhood information.
  • Add more measurement types (temperature, humidity, noise levels).
  • Include historical trends and aggregations (monthly/yearly averages).
