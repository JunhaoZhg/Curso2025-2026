Dataset Analysis for Museums of Madrid Explorer

1. Dataset Analysis

This section analyzes the dataset used for the Museums of Madrid Explorer application, focusing on its schema, quantities, and value ranges. The dataset is sourced from the Madrid City Council's open data portal and contains information about museums in Madrid. It is provided in CSV format with semicolon-separated values, as shown in the provided document "Museos Madrid.csv".

Schema Analysis

The dataset is structured as a CSV file with 29 columns (fields) per row, representing detailed information about each museum. The header row defines the schema as follows:

Fields (Columns):

  • PK: Unique identifier (integer, e.g., 6017808).
  • NOMBRE: Name of the museum (string, e.g., "Casa Museo Fuente del Rey").
  • DESCRIPCION-ENTIDAD: Detailed description of the museum entity (string, often long text).
  • HORARIO: Schedule or opening hours (string, e.g., "De lunes a viernes de 10:00 a 19:00 horas.").
  • EQUIPAMIENTO: Equipment or facilities available (string, often empty or descriptive).
  • TRANSPORTE: Transportation options (string, e.g., "Metro: Antón Martín (línea 1)").
  • DESCRIPCION: Additional description (string, sometimes empty).
  • ACCESIBILIDAD: Accessibility level (string, e.g., "3" or "1,6" indicating codes for accessibility features).
  • CONTENT-URL: URL to more information (string, e.g., "http://www.madrid.es/...").
  • NOMBRE-VIA: Street name (string, e.g., "FUENTE DEL REY").
  • CLASE-VIAL: Street type (string, e.g., "CALLE").
  • TIPO-NUM: Number type (string, e.g., "V").
  • NUM: Street number (string, e.g., "11").
  • PLANTA: Floor (string, often empty).
  • PUERTA: Door (string, often empty).
  • ESCALERAS: Stairs (string, often empty).
  • ORIENTACION: Orientation (string, often empty).
  • LOCALIDAD: Locality (string, e.g., "MADRID").
  • PROVINCIA: Province (string, e.g., "MADRID").
  • CODIGO-POSTAL: Postal code (string, e.g., "28023").
  • COD-BARRIO: Neighborhood code (string, e.g., "7").
  • BARRIO: Neighborhood name (string, e.g., "ARAVACA").
  • COD-DISTRITO: District code (string, e.g., "9").
  • DISTRITO: District name (string, e.g., "MONCLOA-ARAVACA").
  • COORDENADA-X: X coordinate (integer, e.g., "434663").
  • COORDENADA-Y: Y coordinate (integer, e.g., "4478596").
  • LATITUD: Latitude (double, e.g., "40.45546866305681").
  • LONGITUD: Longitude (double, e.g., "-3.7705716814130965").
  • TELEFONO: Phone number (string, e.g., "913 579 116").
  • FAX: Fax number (string, often empty).
  • EMAIL: Email address (string, e.g., "info@fundaciofrandaurel.com").
  • TIPO: Type or category (string, e.g., "/contenido/entidadesYorganismos/FundacionesCulturales").

This schema provides comprehensive details on museum metadata, location, accessibility, and contact information, making it suitable for conversion to Linked Data formats like RDF.

  • Value Ranges:

2. Licensing Analysis

The dataset is sourced from the Madrid City Council’s open data portal ("Datos Abiertos del Ayuntamiento de Madrid"), as indicated by the URLs in the CONTENT-URL field (e.g., "http://www.madrid.es/..."). The analysis of the licensing is as follows:

  • Publisher and Rightsholder: The Madrid City Council (Ayuntamiento de Madrid) is the publisher and rightsholder of the dataset, as it is hosted on their official open data portal.
  • License: The dataset is released under the "Licencia de Uso de Datos Abiertos del Ayuntamiento de Madrid", which is equivalent to the Creative Commons Attribution 4.0 International (CC-BY 4.0) license. This permits free use, redistribution, and adaptation, provided that attribution is given to the source (Madrid City Council). No restrictions on commercial use, but modifications must be shared under the same license terms.
  • Proposed License for Generated Dataset: The generated Linked Data dataset will adopt the Creative Commons Attribution 4.0 International (CC-BY 4.0) license to ensure compatibility with the original dataset’s license and to allow reuse while requiring attribution to the Madrid City Council.

3. Resource Naming Strategy

The resource naming strategy for the ontology and generated Linked Data is defined to ensure consistency, scalability, and compatibility with Linked Data principles. The strategy is as follows:

  • URI Domain: http://data.example.org/madrid/ will be used as the base domain for the ontology and data, following best practices for Linked Data.
  • Ontology URI: The ontology will be defined at http://data.example.org/madrid/museos#.
  • Resource URI Pattern:
    • Museums: http://data.example.org/madrid/museos/resource/{MuseumName}, where {MuseumName} is the name of the museum, we suppose the name is unique (e.g., http://data.example.org/madrid/museos/resource/Casa_Museo_Fuente_del_Rey for Casa Museo Fuente del Rey).
    • Classes: http://data.example.org/madrid/museos#{className}, e.g., ns:Museum, ns:Location.
    • Properties: http://data.example.org/madrid/museos#{propertyName}, e.g., ns:name, ns:hasLocation.
  • Content Negotiation: The URIs will support content negotiation to deliver data in multiple formats:
    • RDF/Turtle (default): text/turtle
    • RDF/XML: application/rdf+xml
    • JSON-LD: application/ld+json
    • HTML: text/html for human-readable representations.
  • Path Structure:
    • Ontology: /madrid/museos#
    • Data instances: /madrid/museos/resource/{PK}
  • Identifier Pattern: Museum identifiers will use the PK field from the CSV to maintain consistency with the source data.
  • Reusability: The URIs are designed to be dereferenceable, allowing external systems to access and link to the museum data. Geospatial properties (e.g., LATITUD, LONGITUD) will map to standard vocabularies like geo:lat and geo:long for compatibility.

Conclusion

The dataset satisfies the requirements for the Museums of Madrid Explorer application, with a well-defined CSV schema and open licensing. The resource naming strategy ensures that the generated Linked Data is consistent, accessible, and reusable, aligning with the principles of Linked Data and the needs of the application.
